# Commitment scheme (§8) — the trust anchor

This is the single source of truth for how CasperProof hashes and commits to agent
input/output. The contract, the attestor, and the verifier MUST agree **byte-for-byte**. The
scheme is implemented once in `packages/commitment` (TypeScript) and mirrored for testing in
`contracts/src/commitment.rs` (Rust). **Never reimplement it inline — import it.**

> The text below is the §8 specification as agreed by every component. The matching code is in
> `packages/commitment/src/{hash,canonical,commitment}.ts` and the Rust mirror in
> `contracts/src/commitment.rs`.

## The scheme

```text
input_hash  = blake2b_256(canonical_bytes(input))
output_hash = blake2b_256(canonical_bytes(output))
commitment  = blake2b_256( input_hash || output_hash || utf8(model_id) || le_u64(timestamp) )
```

- **blake2b-256** (Casper-native, 32-byte digest). **Never sha256.**
  `@noble/hashes` `blake2b(data, { dkLen: 32 })` == Rust `Blake2bVar::new(32)` — verified.
- Preimage field order is fixed: `input_hash` (32 bytes) ‖ `output_hash` (32 bytes) ‖
  `model_id` as UTF-8 bytes ‖ `timestamp` as **little-endian u64** (8 bytes).

## Canonical JSON rules (deterministic, cross-language)

1. **Object keys sorted ascending** (by code point; keys restricted to BMP/ASCII).
   Sorting is recursive at every nesting level.
2. **Compact** — no insignificant whitespace (`{"a":1,"b":2}`, never `{ "a": 1 }`).
3. **UTF-8**; non-ASCII passes through raw (no `\u` escaping) — matches `JSON.stringify`
   and serde_json for BMP characters.
4. **Numbers MUST be integers** (and safe integers). `1.0` serializes differently across
   languages (`"1"` in JS vs `"1.0"` in serde_json). Represent decimals/amounts as
   **strings** (e.g. `"5000000000"`).

> ⚠️ serde_json in this dependency tree is built with `preserve_order` (via casper-types),
> so it keeps insertion order. The Rust canonicalizer therefore sorts keys **explicitly**
> (see `commitment.rs` `write_canonical`). Do not assume serde_json sorts for you.

## TypeScript API (`@casperproof/commitment`)

```ts
import { computeCommitment, hashPayload, canonicalize, verifyOutputHash } from '@casperproof/commitment';

const { inputHash, outputHash, commitment } = computeCommitment({
  input,            // JsonValue
  output,           // JsonValue
  modelId,          // e.g. 'casperproof-riskscorer-v1'
  timestamp,        // unix seconds (u64, safe integer)
});

// verifier: PASS when recomputed output hash equals the on-chain value
const pass = verifyOutputHash(recomputedHex, onchainHex);
```

Helpers exported from the package: `blake2b256`, `toHex`, `fromHex`, `leU64`, `concatBytes`,
`HASH_LENGTH`, `canonicalize`, `canonicalBytes`, `hashPayload`, `commitmentFromHashes`,
`computeCommitment`, `verifyOutputHash`, and the `CanonicalizationError` thrown on
non-integer / unsafe-integer / non-finite numbers.

## TS ⇆ Rust golden-vector parity (non-negotiable)

- `packages/commitment/golden-vectors.json` is generated by `gen:golden` (TS is the generator;
  see `packages/commitment/src/gen-golden.ts`). The Rust test
  `commitment::golden_parity::rust_matches_typescript_golden_vectors` reads the **same file** via
  `include_str!("../../packages/commitment/golden-vectors.json")` and asserts identical hashes.
- The Rust mirror re-derives the scheme in exactly one place — that host-only `#[cfg(test)]`
  module — purely to prove agreement. The deployed contract never recomputes hashes; it stores
  and compares the bytes the off-chain components produced.
- CI runs both `pnpm --filter @casperproof/commitment test` and `cargo test --lib`, so any
  divergence fails the build. The Rust side also pins two standard BLAKE2b-256 vectors that must
  equal the TS `hash.test.ts` values:
  - `blake2b_256("")  = 0e5751c026e543b2e8ab2eb06099daa1d1e5df47778f7787faab45cdf12fe3a8`
  - `blake2b_256("abc") = bddd813c634239723171ef3fee98579b94964e3bb1cb3e427262c8c068d52319`
- If you change canonicalization or the preimage layout, regenerate vectors **and** keep the Rust
  mirror in sync, or the parity test breaks. That break is intentional — fix both sides.

Regenerate the vectors with:

```bash
pnpm --filter @casperproof/commitment gen:golden
```

## Worked example (from the committed golden vectors)

Taken verbatim from `packages/commitment/golden-vectors.json` (vector `key-order-insensitive`).
It exercises the two subtle rules: recursive key sorting and compact output.

Inputs:

```jsonc
input    = { "b": 2, "a": 1, "c": { "z": 26, "a": 1 } }
output   = { "score": 42, "tier": "LOW" }
modelId  = "casperproof-riskscorer-v1"
timestamp = 1718600000
```

Step 1 — canonicalize (keys sorted recursively, compact):

```text
canonicalInput  = {"a":1,"b":2,"c":{"a":1,"z":26}}
canonicalOutput = {"score":42,"tier":"LOW"}
```

Step 2 — hash the canonical bytes with blake2b-256:

```text
input_hash  = 39495ec0f114bfd99dda81e629b6851bc878ff704fb1e2c2286ec42233bbb1d7
output_hash = 0aa28e94eaa6845edec84af8a6c77121a0a3f21815ff09c49733f04af52f5e9e
```

Step 3 — build the commitment preimage and hash it:

```text
preimage    = input_hash (32B) ‖ output_hash (32B) ‖ utf8("casperproof-riskscorer-v1") ‖ le_u64(1718600000)
commitment  = blake2b_256(preimage)
            = 9fe2649fbafcee3d8cb871d070c38f9ec6b023fcfc9cdea9a766e4a785d08f01
```

Other committed vectors cover empty objects, the risk-scorer and claim-oracle output shapes,
BMP unicode (`café`, `漢字`, `verified ✓`), booleans/null, integer arrays, and nested
arrays/objects. The simplest reference value:

```text
canonical({}) = "{}"
blake2b_256("{}") = c09da522dac261c3d2566230bed10d2f3ef13f8e7654576c12e857e07f786098
```

## Verification (PASS / FAIL)

The verifier loads the attestation by id, fetches the payload by `uri`, recomputes
`output_hash'` from the payload via this package, and asserts `output_hash' == onchain.output_hash`.

- **Match ⇒ PASS** — the off-chain payload is exactly what was attested.
- **Mismatch ⇒ FAIL** — the payload was tampered with. This is a normal verification result, not
  an exception; it surfaces over HTTP/MCP as the RFC 7807 `TAMPERED_PAYLOAD` problem
  (HTTP 422, `valid: false`, both hashes attached). See [`CONTRACTS.md`](./CONTRACTS.md).

Because the store key is the blake2b-256 of the payload bytes (content addressing), changing a
byte changes the key — so a tamper at the same URI is exactly what the slash demo exploits, and
the verifier catches it.
