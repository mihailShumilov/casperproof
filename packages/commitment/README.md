# @casperproof/commitment

The CasperProof **trust anchor (§8)**: the `blake2b-256` commitment scheme with deterministic
canonical JSON. Implemented once here (TypeScript) and mirrored for testing in
`contracts/src/commitment.rs` (Rust). The contract, the attestor, and the verifier MUST agree
**byte-for-byte** — so every component imports this package and **never reimplements the scheme
inline**.

See the full specification, rules, and worked example in [`../../docs/COMMITMENT.md`](../../docs/COMMITMENT.md).

## The scheme

```text
input_hash  = blake2b_256(canonical_bytes(input))
output_hash = blake2b_256(canonical_bytes(output))
commitment  = blake2b_256( input_hash || output_hash || utf8(model_id) || le_u64(timestamp) )
```

- **blake2b-256** (Casper-native, 32-byte digest) via `@noble/hashes` — never sha256.
- Preimage field order is fixed: `input_hash` (32B) ‖ `output_hash` (32B) ‖ `model_id` (UTF-8) ‖
  `timestamp` (**little-endian u64**, 8B).

## Canonical JSON rules

1. Object keys **sorted ascending** (by code point; BMP/ASCII), recursively at every level.
2. **Compact** — no insignificant whitespace.
3. **UTF-8**; non-ASCII passes through raw (matches `JSON.stringify`).
4. Numbers MUST be **integers** (and safe integers); represent decimals/amounts as **strings**
   (e.g. `"5000000000"`). Non-integer / unsafe / non-finite numbers throw `CanonicalizationError`.

## Quick start

```ts
import {
  computeCommitment,
  hashPayload,
  canonicalize,
  verifyOutputHash,
} from '@casperproof/commitment';

const { inputHash, outputHash, commitment } = computeCommitment({
  input: { address: 'account-hash-aabbcc', signals: 15 },
  output: { score: 73, tier: 'HIGH' },
  modelId: 'casperproof-riskscorer-v1',
  timestamp: 1718600123, // unix seconds (u64, safe integer)
});

// verifier: PASS when the recomputed output hash equals the on-chain value
const pass = verifyOutputHash(hashPayload({ score: 73, tier: 'HIGH' }), outputHash);

canonicalize({ b: 2, a: 1 }); // '{"a":1,"b":2}'
```

## Golden vectors & TS⇆Rust parity

`golden-vectors.json` is generated by TypeScript and read by the Rust test
`commitment::golden_parity::rust_matches_typescript_golden_vectors` (via `include_str!`), which
asserts identical canonicalization and hashes. CI runs both sides; any divergence fails the build.

```bash
pnpm --filter @casperproof/commitment gen:golden   # regenerate the vectors (TS is the generator)
```

If you change canonicalization or the preimage layout, regenerate the vectors **and** keep the
Rust mirror in sync — the parity break is intentional, fix both sides.
